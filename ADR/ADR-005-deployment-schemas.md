# ADR-005: Deployment Request, Resolved Record, and Canonical Schemas

> **Status**: `ACCEPTED`  
> **Date**: 2026-06-23  
> **Revised**: 2026-06-23 (incorporates mandatory review changes)  
> **Decision makers**: César Guzmán  
> **Scope**: Scanalyze Dedicated Deployment Platform  
> **Depends on**: ADR-001, ADR-002

---

## Context

El plan original usaba un único YAML como "deployment contract". Esto no funciona porque los mismos campos no pueden estar presentes antes y después de provisionar una cuenta.

Además, el release manifest necesita evidencia de supply chain más allá de digests, y los schemas deben ser estrictos para prevenir errores silenciosos.

---

## Decision

### Dos Contratos Separados

| Contrato | Propósito | Quién lo crea | Lifecycle |
|---|---|---|---|
| **Deployment Request** | Input para provisioning | Humano (via PR) | GitOps, versionado |
| **Resolved Deployment Record** | Estado real post-provisioning | Plataforma (automated) | Generado, no editado libremente |

## Schema 1: Deployment Request (`scanalyze.deployment-request.v1`)

```yaml
# deployments/acme-corp-prod/request.yaml
schema_version: scanalyze.deployment-request.v1

request:
  request_id: req_01J5ABCDEF1234567890  # ULID-based
  customer_id: acme-corp
  deployment_id: acme-corp-prod-us-east-1
  environment: prod
  account_ownership: scanalyze-managed    # customer-managed → REJECTED in v1
  primary_region: us-east-1
  release_version: "2026.06.0"

commercial:
  contract_ref: ctr_01J5ABCDEF0000000001
  deployment_profile: enterprise-standard
  support_profile: standard
  features:
    employee_profiles: true
    cost_analytics: true
    contract_publisher: true

capacity:
  profile: enterprise-standard

domains:
  api: api.acme-corp.scanalyze.cloud
  frontend: app.acme-corp.scanalyze.cloud

security:
  data_classification: restricted
  retention_profile: mexico-enterprise-7y
  deletion_protection: true
  backup_profile: enterprise

resilience:
  dr_profile: backup-restore
  recovery_region: null

processing_domains:
  - bank
  - personal
  - gov
```

### Key design decisions

| Decision | Rationale |
|---|---|
| **ULID-based request_id** (`req_<ULID>`) | No collision between pipelines/branches; sortable by time |
| **`additionalProperties: false`** everywhere | Typos like `deletion_protectionn: true` are rejected, not ignored |
| **`processing_domains`** not `document_domains` | Clarifies routing purpose; `platform` is mandatory infrastructure, not a domain |
| **`account_ownership: customer-managed` → REJECTED** | Schema-compatible but `FEATURE_NOT_IMPLEMENTED` in v1 workflow |
| **`retention_profile`** not `retention_days` | Profile resolves S3, DDB, logs, audit, legal hold, KMS lifecycle |
| **`contract_ref`** not detailed commercial data | Deployment repo is not the contractual system of record |
| **No `account_id` or `deployment_role_arn`** | Don't exist yet at request time |

### Integrity fields (generated by tooling, not user)

The platform generates and attaches these at submission:

| Field | Value |
|---|---|
| `request_digest` | SHA-256 of canonical serialization |
| `schema_digest` | SHA-256 of the JSON Schema used for validation |
| `approval_record` | PR approval reference |
| `source_commit` | Git commit SHA |

### Validation rules

1. Schema validation with `additionalProperties: false` recursively
2. `primary_region` must exist in the region capability matrix
3. `release_version` must be supported in that region
4. `account_ownership: customer-managed` → reject with `FEATURE_NOT_IMPLEMENTED`
5. `processing_domains` must contain only `bank`, `personal`, `gov`
6. Domain DNS ownership must be validated
7. Domain uniqueness must be checked against the registry
8. `customer_id` and `deployment_id` must be unique in the registry
9. `retention_profile` must exist in the retention profiles catalog
10. `deployment_profile` must exist in the deployment profiles catalog

### Status State Machine

```
REQUESTED → PROVISIONING
PROVISIONING → READY | PROVISIONING_FAILED
PROVISIONING_FAILED → PROVISIONING (retry)
READY → DEPLOYING
DEPLOYING → ACTIVE | DEPLOYMENT_FAILED
DEPLOYMENT_FAILED → DEPLOYING (retry) | SUSPENDING
ACTIVE → UPGRADING | SUSPENDING | OFFBOARDING
UPGRADING → ACTIVE | ROLLBACK_REQUIRED
ROLLBACK_REQUIRED → UPGRADING (rollback) | SUSPENDING
SUSPENDING → SUSPENDED
SUSPENDED → DEPLOYING (reactivation) | OFFBOARDING
OFFBOARDING → ARCHIVED
```

All status transitions use **conditional writes** (DynamoDB `ConditionExpression`). State transition history is **append-only**.

### Release Attestation (`scanalyze.release-attestation.v1`)

Separate from the manifest. Generated and signed by the release pipeline:

```json
{
  "schema_version": "scanalyze.release-attestation.v1",
  "release_version": "2026.06.0",
  "manifest_digest": "sha256:...",
  "attestation_digest": "sha256:...",
  "attestation_signature": "arn:aws:signer:...",
  "attested_at": "2026-06-23T00:30:00Z",
  "attested_by": "release-pipeline-v1.2.0",
  "checks": {
    "all_images_scanned": true,
    "all_images_signed": true,
    "all_sboms_present": true,
    "no_critical_findings": true,
    "no_unwaivered_high_findings": true,
    "source_digest_equals_destination_digest": true,
    "manifest_signed": true,
    "toolchain_versions_locked": true,
    "module_digests_verified": true,
    "release_authorized": true
  },
  "authorization": {
    "approver": "release-approver-identity",
    "approval_id": "approval-01J5...",
    "approved_at": "2026-06-22T23:00:00Z"
```

The attestation **references** the manifest digest. It is not self-affirming booleans inside the manifest.

## Schema 4: Region Capability Matrix

```yaml
schema_version: scanalyze.region-capability.v1
region: us-east-1
status: candidate    # → 'supported' only after runtime validation

release_version: "2026.06.0"
validated_at: null
validation_execution_id: null
validator_version: null
evidence_digest: null
expires_at: null

bedrock:
  runtime_available: true
  models:
    amazon.nova-lite-v1:0:
      status: candidate
      access_required: true
      access_mode: on-demand
    anthropic.claude-3-haiku-20240307-v1:0:
      status: candidate
      access_required: true
      access_mode: on-demand

textract:
  document_text_detection: candidate
  document_analysis: candidate
  expense_analysis: candidate

required_service_quotas:
  ecs_tasks_per_cluster: 100
  ecr_repositories: 10
  sqs_queues: 20
  dynamodb_tables: 20

known_limitations: []
```

### Validation rules

- `status: candidate` until runtime evidence exists
- `status: supported` requires `validated_at`, `validation_execution_id`, `evidence_digest`
- `expires_at` forces periodic revalidation
- Region capability certifies **regional** capacity; account-level preflight validates **account-specific** capacity (model access, quotas, SCPs)
- Bedrock model access can depend on account configuration; the matrix certifies the region, the preflight validates the account

## Schema 5: Inter-Layer Contract Envelope

```yaml
# SSM parameter: /scanalyze/deployments/{deployment_id}/contracts/{layer}/v{version}
schema_version: scanalyze.contract.v1

contract:
  layer: network
  version: 1
  deployment_id: acme-corp-prod-us-east-1
  account_id: "123456789012"
  region: us-east-1
  producer_release: "2026.06.0"
  producer_module: network
  producer_module_version: "2.1.0"
  contract_digest: "sha256:..."
  generated_at: "2026-06-23T18:30:00Z"
  output_schema_version: "scanalyze.contract-network.v1"

outputs:
  vpc_id: vpc-0abc123...
  private_subnet_ids: [subnet-0abc..., subnet-0def...]
  public_subnet_ids: [subnet-0ghi..., subnet-0jkl...]
  nat_gateway_ids: [nat-0abc...]
```

### Size strategy

| Contract size | Storage |
|---|---|
| Small (< 4KB) | JSON in single SSM parameter |
| Large (> 4KB) | Small manifest in SSM with `payload_uri` and `payload_digest` pointing to S3 |

### Consumer validation (fail-closed)

Consumers **must fail** if any of these checks fail:

1. Unknown critical fields present
2. `contract_digest` does not match computed digest
3. `account_id` does not match current account
4. `deployment_id` does not match current deployment
5. `region` does not match current region
6. `output_schema_version` is incompatible

## Observability Export Schema

### Allowed to exit customer account (toward Shared Services / Operations)

| Dimension/Metric | Notes |
|---|---|
| `deployment_id` | **Primary dimension** — resolve `customer_id` via registry |
| `account_id` | For cross-account correlation |
| `region` | For regional aggregation |
| `release_version` | For release tracking |
| Service health (ECS desired/running) | CloudWatch metric |
| Alarm state changes | SNS / EventBridge event |
| Token counts (aggregate, no per-document) | Metric |
| Aggregate costs | Billing data |
| Pipeline/deployment status | Event |
| Resource inventory (non-sensitive) | Metadata |

### Prohibited from exiting

| Data | Reason |
|---|---|
| OCR text / extracted content | PII |
| Document names / S3 keys with PII | PII |
| Prompts | Customer data |
| LLM responses | Customer data |
| JWTs | Credentials |
| Personal fields (CURP, RFC, NSS, CLABE) | PII |
| Bank account data | PII |
| Raw exception stack traces | May contain PII (even within customer account — sanitize) |
| `document_id` as CloudWatch dimension | Cardinality + PII risk |
| `batch_id` or `invocation_id` as metric dimensions | Cardinality |

### Export controls

- Retention and encryption mandatory on all exported data
- Unknown fields rejected by the export schema
- **Sentinel tests for PII leakage** — automated tests that inject known PII patterns and verify they do not appear in exported metrics/logs
- Export schema versioned independently

## Consequences

### Positivas
- Separación clara de input (request) vs output (record)
- Schemas validables y estrictos
- Release manifests con full supply chain evidence
- Attestation separada, no booleans autoafirmados
- Region capability previene deployments en regiones no certificadas
- Inter-layer contracts elimina dependencias implícitas

### Negativas
- Más schemas que mantener
- Validación adicional en cada paso
- Region capability matrix requiere actualización y validación periódica
- Schema migration requires formal process
